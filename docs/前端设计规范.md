## 1. 技术栈与目录

技术栈与目录结构**完全以 [`README.md`](../README.md) 为准**，本文不重复，只强调三条贯穿全篇的硬约束：

1. **状态分治（头号规范）**：服务端数据（列表/详情/统计）一律走 **TanStack Query**；Zustand 只扛纯 UI 态（侧栏折叠/主题/语言/token）。绝不把列表数据塞进 Zustand。
2. **类型不手抄**：后端出入参类型由 `pnpm codegen` 从 `janus-server` 的 OpenAPI 生成到 `src/api/generated/`，禁手改、禁手抄。
3. **feature 边界由 ESLint 强制**：`import/no-restricted-paths` 划线，跨 feature 复用一律下沉到 `components/hooks/utils`，违例 `pnpm lint` 直接红。

> **codegen 是契约同步的唯一可信源。** 本文 §5–§8 列出的所有结构（信封、分页、错误码、枚举）都应能在 `generated/types.ts` 里找到对应类型；若手写类型与 generated 冲突，以 generated 为准并修正本文。

## 2. 视觉系统

取自 Aether v2.0 设计稿的视觉语言，落到 AntD 5 的 `ConfigProvider` 主题 token。**这是设计稿里可直接复用的部分**（颜色/间距/质感与后端无关）。

### 2.1 色彩

| 角色 | 色值 | 用途 |
|---|---|---|
| 主色 Laplace Blue | `#192E76` | 主按钮、链接、选中态、关键数据强调 |
| 侧栏深底 | `#142B70` ~ `#17327C` | 左侧导航背景（深海军蓝） |
| 成功 / 在线 | `#67C23A` | 正常状态徽标、健康指示、active |
| 警告 | `#E6A23C` | 临界配额、降级、需关注 |
| 危险 / 错误 | `#F56C6C` | 失败状态、删除、超额、disabled |
| 页面底色 | `#F5F6FA` 级浅灰 | 内容区背景（衬托白卡片） |
| 卡片 / 表格底 | `#FFFFFF` | 所有 ProCard / ProTable 容器 |

```typescript
// src/styles/theme.ts —— 喂给 <ConfigProvider theme={...}>
export const antdTheme = {
  token: {
    colorPrimary: '#192E76',
    colorSuccess: '#67C23A',
    colorWarning: '#E6A23C',
    colorError:   '#F56C6C',
```

> **颜色语义统一走 token**，组件里不写裸 hex。状态徽标颜色由 §8 的 `valueEnum` 统一映射，不在组件里硬编码。

### 2.2 间距与圆角

- 基础间距栅格 **8px**（4/8/16/24）；卡片内边距 24px，卡片间距 16px。
- 圆角统一 6px（卡片/按钮/输入框）；徽标/Tag 用胶囊圆角。
- 表格走 AntD `size="small"` 的紧凑密度（内部后台数据密集，优先信息密度）。

### 2.3 字体与字号

- 字体栈：系统默认（`-apple-system, "Segoe UI", "PingFang SC", "Microsoft YaHei", sans-serif`），不引 web font（内网、无 SEO、省加载）。
- 字号：正文 14px、表格 14px、卡片标题 16px、KPI 数字 28–32px 加粗、辅助说明 12px 次级灰。
- 数字（金额/用量/token 数）用**等宽对齐右对齐**，便于列内比对。

### 2.4 质感

- **白卡片浮于浅灰底**，卡片用极轻投影（`0 1px 2px rgba(0,0,0,.06)`），不堆重阴影。
- 表格行 hover 浅色高亮；选中行主色淡背景。
- 详情/编辑用**右侧抽屉（Drawer）+ 半透明遮罩**，不用全屏跳转（保留列表上下文）。

## 3. 布局规范

整体采用 AntD Pro 的 **`ProLayout`**（侧栏 + 顶栏 + 内容区），骨架对齐 Aether 稿。

### 3.1 框架结构

```
┌────────────┬───────────────────────────────────────┐
│            │  顶栏：面包屑 / 占位 ··· 语言切换 用户菜单 │
│  侧栏       ├───────────────────────────────────────┤
│  深海军蓝   │                                       │
│  ~250px    │   内容区（浅灰底）                       │
│  可折叠     │   └─ 白卡片承载 ProTable / ProForm      │
│            │                                       │
└────────────┴───────────────────────────────────────┘
```

- **侧栏**：固定左侧，展开 ~250px、折叠 ~80px（仅图标）；折叠态存 `ui.store.ts`（Zustand）。菜单由权限驱动（§4.3）。
- **顶栏**：右侧放语言切换 + 用户菜单（`RightContent.tsx`）；左侧面包屑由路由 meta 生成。
- **内容区**：浅灰底，业务内容一律装进白卡片（`PageContainer` + `ProCard`）。

### 3.2 页面骨架模板

| 页型 | 骨架 | 代表页 |
|---|---|---|
| **仪表盘** | 顶部 4 KPI 卡一行 → 健康条 → 分析/负载图 → 趋势/配额/异常 | 概览 |
| **列表页** | 单张白卡片：标题 + 计数徽标（左上）+ 工具栏（右上）+ ProTable | 用户/凭证/目录/用量 |
| **主从页** | 左侧组织树面板 + 右侧 ProTable + 右抽屉详情 | 用户管理 |
| **配置页** | ProForm 分组卡片 | 配额/系统配置 |

> 列表页**不做整页跳转编辑**：新建/编辑/查看详情统一走右侧 Drawer（§9.3），点击行不离开列表。

## 4. 领域与路由

领域划分**严格对齐后端 `janus-server` 的真实模块**（不是 Aether 原型的臆测领域）。features 目录与下表一一对应。

### 4.1 业务模块（对齐后端 + README features/）

| feature | 领域 | 后端依据 |
|---|---|---|
| `users` | 用户 / 角色 / 部门管理（RBAC） | `admin` 用户体系 + RuoYi 式菜单权限 |
| `credentials` | API Key（`sk-key`）管理 | `ApiKeyStatus`，网关协议侧凭证 |
| `catalog` | 逻辑模型 / 上游渠道 / 号池 | `upstream_channel` → `channel_key` → `logical_model` → `model_deployment` 链 |
| `grants` | 模型分配（谁能用哪些逻辑模型） | `user_model_grant`，`GrantScope = user/department` |
| `usage` | 用量统计 / 内部账单（图表 + ProTable） | `usage_record`，`UsageStatus = success/error/timeout` |
| `quota` | 配额配置 | `global/user/department × metric × period`，见 §4.2 |

### 4.2 配额模型（按真实后端，**非** Aether 的 `quota_policy`）

配额维度由后端枚举定义，前端表单/筛选直接映射：

- **Scope**（`QuotaScope`）：`user` / `department` / `global`
- **Metric**（`QuotaMetric`）：`tokens` / `requests` / `cost`
- **Period**（`QuotaPeriod`）：`daily` / `monthly` / `total`

配额是 `scope × metric × period` 的组合（后端用 NULLS NOT DISTINCT + CHECK 约束保证唯一），前端配额表单按这三个维度组织，不要发明 Aether 稿里的「策略对象」。

### 4.3 路由与权限

- 路由树在 `app/router/`，私有路由统一过 `AuthGuard`（验 token）+ `PermissionGuard`（验权限码）。
- **权限码是 RuoYi 风格的菜单权限串**：如 `system:user:list`、`system:user:add`。菜单类型由后端 `MenuType` 决定：
  - `catalog` —— 分组节点，无路由页（只做菜单分组）
  - `menu` —— 可路由页面节点
  - `button` —— 细粒度操作权限（控制按钮显隐）
- 菜单/按钮显隐由后端下发的权限集驱动；前端 `PermissionGuard` 拦路由、`<Access>` 包裹按钮。
- 路由路径常量集中在 `app/router/paths.ts`，组件里用 `useNavigate(paths.xxx)`，不写裸路径串。

### 4.4 认证（双轨）

后端是**双认证体系**，前端只处理管理面那一轨：

- **管理后台（本前端）**：RS256 JWT（access + refresh，refresh 轮换）。token 存 `auth.store.ts`，Axios 拦截器挂 `Authorization: Bearer`。
- **网关协议端点（LLM/MCP）**：`sk-key`（API Key），由 `credentials` feature **管理**，但前端自身不用它发管理面请求。

> 401/`auth.invalid_token` / `auth.token_revoked` → 触发 refresh 或踢回登录页；`auth.refresh_invalid` → 直接登出。

## 5. API 契约（真实 wire format）

> ⚠️ **重要更正**：README §目录注释与 i18n 段称错误响应是 **RFC 9457 `problem+json`**，这与后端实现**不符**。真实后端（[`src/responses.py`](../../janus-server/src/responses.py) / [`src/exceptions.py`](../../janus-server/src/exceptions.py)）采用 **方案 B：以 `success` 布尔字段判别的同构信封**，**不是** problem+json。**以下结构以后端源码为准**，Axios 拦截器与 codegen 据此实现。

### 5.1 成功信封 `SuccessEnvelope<T>`

```jsonc
// HTTP 2xx
{
  "success": true,
  "data": <T>,          // 业务负载（可能为 null）
  "trace_id": "..."     // 链路 ID，与响应头 X-Request-ID 一致
}
```

### 5.2 错误信封（与成功**同构**，靠 `success` 判别）

```jsonc
// HTTP 4xx / 5xx
{
  "success": false,
  "code": "auth.invalid_token",   // 机器可读稳定错误码（§6），前端据此查文案
  "params": { "used": 100, "limit": 100, "period": "daily" },  // 结构化插值参数
  "trace_id": "...",
  "errors": [                      // 仅入参校验失败（422）时出现
    { "field": "email", "type": "value_error", "msg": "..." }
  ]
}
```

**设计要点（贴合后端注释）**：

- 用 `success: boolean` 做判别字段，前端 TS **discriminated union** 收窄干净，不靠解析 code 判成败。
- 错误码字段名是 **`code`**（不是 `error_code`）——对齐 JSON:API / Stripe / GitHub 惯例。
- **走纯 code 的 i18n 路线**：后端只发机器可读 `code` + `params`，**绝不发 `message`**。展示文本完全由前端拥有（§7）。
- HTTP 状态码语义保留（成功 2xx / 错误 4xx/5xx），前端按状态码分流，body 仅承载细节。
- 网关面（OpenAI/Anthropic 兼容端点）**不使用本信封**——它透传上游原生协议；本前端不直接消费网关面。

### 5.3 前端类型（TS discriminated union）

```typescript
// src/types/api.ts —— 与 generated 对齐（codegen 产物为准）
export interface SuccessEnvelope<T> {
  success: true;
  data: T | null;
  trace_id: string;
}

export interface ErrorEnvelope {
  success: false;
  code: string;                          // 见 §6 ErrorCode
  params: Record<string, unknown>;
  trace_id: string;
  errors?: { field: string; type: string; msg: string }[];
}

export type ApiResponse<T> = SuccessEnvelope<T> | ErrorEnvelope;
```

### 5.4 分页信封 `Page<T>`（offset/limit）

列表端点统一返回 `Page<T>`，**offset/limit 分页**（非 page/pageSize）：

```jsonc
{
  "items": [ /* T[] */ ],
  "total": 128,
  "limit": 20,
  "offset": 0
}
```

```typescript
export interface Page<T> {
  items: T[];
  total: number;
  limit: number;   // ≥ 1
  offset: number;  // ≥ 0
}
```

> **ProTable 适配**：ProTable 默认是 `current`/`pageSize` 思维，request 适配器要做 `offset = (current - 1) * pageSize`、`limit = pageSize` 的换算，返回 `{ data: items, total, success: true }`。这层换算封装进 `shared` 的 `proTableRequest` 工具，各 feature 不重复写。
>
> **树形数据无分页**：组织/菜单树等返回完整树结构，不套 `Page<T>`。

### 5.5 Axios 拦截器职责

```
请求拦截：挂 Authorization: Bearer <jwt>；挂 Accept-Language（当前 locale）
响应拦截：
  - success === true  → 透出 data
  - success === false → 按 code 查 i18n（§7）+ params 插值 → toast / 表单级提示
                        401 类 → 触发 refresh 或登出
                        422 + errors → 映射到 ProForm 字段级错误
  - trace_id 统一记进错误日志（便于对后端日志排查）
```

## 6. 错误码目录

后端 `ErrorCode`（[`src/enums.py`](../../janus-server/src/enums.py)）共 **23 个**稳定 code。前端在 `locales/{lang}/error.ts` 按 code 建 key，用 `params` 插值。**这是错误文案的完整清单，不增不减**（增删以后端为准）。

| code | HTTP（典型） | 含义 | 建议文案方向 |
|---|---|---|---|
| `auth.invalid_token` | 401 | JWT 或 sk-key 无效/过期 | 登录已失效，请重新登录 |
| `auth.token_revoked` | 401 | 会话被吊销（登出/踢下线/重用） | 会话已失效 |
| `auth.refresh_invalid` | 401 | refresh token 未知/过期/已轮换 | 请重新登录 |
| `auth.account_locked` | 403 | 登录失败过多被锁 | 账号已锁定，请稍后再试 |
| `auth.password_too_weak` | 400 | 新密码强度不足 | 密码强度不足 |
| `auth.user_disabled` | 403 | 已认证用户被禁用 | 账号已禁用 |
| `auth.forbidden` | 403 | 已认证但无权限 | 无权限执行此操作 |
| `attach.not_found` | 404 | 附件不存在 | 附件不存在 |
| `attach.invalid_image` | 400 | 上传文件非可解码图片 | 图片格式无法识别 |
| `attach.too_large` | 400 | 上传超大小限制 | 文件超出大小限制 |
| `model.not_granted` | 403 | 主体未被授予该逻辑模型 | 未获该模型使用权限 |
| `model.not_found` | 404 | 逻辑模型不存在/已禁用 | 模型不存在或已禁用 |
| `model.unavailable` | 503 | 逻辑模型无可用部署 | 模型当前不可用 |
| `model.no_available_channel` | 503 | 逻辑模型无可用渠道 | 模型无可用渠道 |
| `quota.exceeded` | 429 | 配额耗尽 | 已用 {used}/{limit}，{period} 配额耗尽 |
| `rate_limit.exceeded` | 429 | 硬限流（RPM/TPM/并发）触发 | 请求过于频繁，请稍后再试 |
| `upstream.error` | 502 | 上游返回错误 | 上游服务异常 |
| `upstream.timeout` | 504 | 上游超时 | 上游响应超时 |
| `upstream.rate_limited` | 429 | 上游号池被限流 | 上游限流，请稍后再试 |
| `request.invalid` | 400/404/422 | 入参/参数无效（含路由不存在、方法不允许） | 请求参数有误 |
| `request.conflict` | 409 | 写入违反唯一/外键约束 | 已存在或存在冲突 |
| `service.unavailable` | 503 | 后端依赖（Redis）不可用，瞬时可重试 | 服务暂不可用，请重试 |
| `internal.error` | 500 | 未预期平台错误 | 系统错误，请联系管理员 |

> `params` 里的插值键（如 `quota.exceeded` 的 `used`/`limit`/`period`）由后端抛错时塞入；前端文案模板用 `{used}` 占位，由 react-intl 插值。`422` 的字段级错误从 `errors[].field` 映射到对应 ProForm 字段。

## 7. i18n（与后端契约，G16）

**核心契约：后端只发稳定标识符，前端拥有全部展示文本。**

| 文本类别 | 后端发什么 | 前端怎么渲染 |
|---|---|---|
| **枚举值**（状态/类型） | `StrEnum` 的 code（如 `"active"`） | ProComponents `valueEnum` 映射本地化 label + 颜色（§8） |
| **业务错误** | 信封的 `code` + `params`（§5.2/§6） | 按 `code` 查 `locales/*/error.ts`，用 `params` 插值 |
| **菜单/按钮/字段 label** | （前端自有） | i18n 静态资源，`menu.system.user` 两层命名空间 |

实现要点：

- 用 **react-intl** + AntD `ConfigProvider` 的 `locale`（zh-CN / en-US 同步切换 AntD 内置文案）。
- 请求头带 `Accept-Language`（后端日志/审计可能用，但**展示文本不依赖后端**）。
- 当前语言存 `locale.store.ts`（Zustand）。
- `error.ts` 的 key **严格对齐 §6 的 23 个 code**；新增 code 必须同步补文案，缺 key 时回落到通用「操作失败」+ 上报。
- **枚举 label 的可信源在后端** `locales/{lang}/enums.json`（若后端提供），前端理想做法是 codegen 同步而非手抄，避免漂移；当前以 §8 表为准手维护，后端补 enums 资源后切 codegen。

## 8. 枚举目录与 valueEnum

后端枚举（`StrEnum`，[`src/enums.py`](../../janus-server/src/enums.py)）。前端**不手抄字面量**——类型来自 codegen；下表给出**展示映射**（label + 状态色），统一封装成可复用的 `valueEnum`。

```typescript
// 示例：src/features/users/components/userStatusValueEnum.ts
import type { ProSchemaValueEnumMap } from '@ant-design/pro-components';
// label 走 intl，这里用 key；颜色对齐 §2.1 token 语义
export const userStatusValueEnum: ProSchemaValueEnumMap = new Map([
  ['active',   { text: 'enum.userStatus.active',   status: 'Success' }],
  ['disabled', { text: 'enum.userStatus.disabled', status: 'Error' }],
]);
```

| 枚举 | 值 | 用途 | 建议状态色 |
|---|---|---|---|
| `ActiveStatus` | `active` / `disabled` | 配置记录启用态 | Success / Default |
| `UserStatus` | `active` / `disabled` | 员工用户状态 | Success / Error |
| `ApiKeyStatus` | `active` / `disabled` | API Key 是否可认证 | Success / Error |
| `ChannelStatus` | `active` / `disabled` | 渠道是否参与路由 | Success / Default |

## 9. 组件模式

### 9.1 列表页 = ProTable

- 一页一张白卡片，标题 + 计数徽标在左上、工具栏（新建/批量/导出/刷新）在右上。
- **数据走 TanStack Query**，ProTable 的 `request` 适配 `Page<T>`（§5.4 的 offset/limit 换算）。
- 筛选条件进 **query key**（`userQueries.list(filters)`），筛选变化自动重取，缓存命中即秒开。
- 列的状态/类型字段用 §8 的 `valueEnum` 渲染；操作列用 `<Access>` 按权限码（§4.3）控制按钮显隐。

```typescript
// 形态示意（真实代码以 feature 内为准）
<ProTable<User>
  request={proTableRequest(userQueries.list)}  // 封装 offset/limit 换算 + Page<T> 适配
  columns={userColumns}                        // 含 valueEnum 状态列
  rowKey="id"
  search={{ filterType: 'light' }}
  toolBarRender={() => [<NewUserButton key="new" perm="system:user:add" />]}
/>
```

### 9.2 表单 = ProForm

- 新建/编辑用 **ProForm / ProFormXxx**（联动、校验、分步开箱）。
- 提交走 `useMutation`，成功后 `invalidateQueries({ queryKey: userQueries.lists() })` 级联失效，列表自动刷新。
- **422 字段级错误**：从错误信封 `errors[].field` 映射到对应 ProForm 字段，`form.setFields([{ name, errors }])`。

### 9.3 详情/编辑 = 右侧 Drawer

- 列表行操作打开**右侧 Drawer + 半透明遮罩**（§2.4 / §3.2），不整页跳转、不丢列表上下文。
- Drawer 内详情数据走 `userQueries.detail(id)`（独立 query，可与列表共享缓存）。

### 9.4 主从页 = 组织树 + 表格

- 用户管理等：左侧组织/部门树（树形数据无分页，§5.4），选中节点作为筛选条件进表格 query key，右侧 ProTable 联动。

### 9.5 仪表盘 = KPI 卡 + 图表

- 顶部 4 张 `StatCard`（KPI），下方健康条 + 分析/负载/趋势图。
- 每个数据块**独立 query**（独立 staleTime / 失效），互不阻塞渲染；图表库与 AntD Pro 配套（`@ant-design/charts`）。

## 10. 原型对齐与裁剪（必读）

Aether v2.0 是基于 React 18 / Vite 5 + mock 数据的早期原型稿，其中**部分领域/表结构在真实 `janus-server` 中不存在**。前端**只取其视觉/交互语言**，领域与数据结构**一律以后端为准**。以下是明确的裁剪清单——**不要为这些臆测结构建 feature/页面/类型**：

| Aether 稿里的东西 | 真实情况 | 前端处置 |
|---|---|---|
| `mcp_tool` / `mcp_call_log` 表、MCP 工具注册页 | 平台**本身就是 MCP server**，工具是代码、**零新增表** | **不建** MCP 工具管理 feature |
| `skill` / `skill_targets` 表、技能管理 | 属 **Hermes 客户端域**，非 janus-server | **不建**（归 Hermes 端，不在本前端） |
| `client_version`、版本分发页 | 属 **Hermes 客户端域** | **不建** |
| `metrics_daily` 预聚合表 | 后端无此表；用量来自 `usage_record` 聚合 | 统计走 `usage` 真实端点，不假设预聚合表 |
| `quota_policy` 策略对象 | 真实模型是 `scope × metric × period`（§4.2） | 配额表单按三维度组织，不造「策略对象」 |
| RFC 9457 `problem+json` 错误体 | 真实是**方案 B 同构信封**（§5.2） | 拦截器按 `success` 判别，**非** problem+json |
| `page` / `pageSize` 分页 | 真实是 **offset/limit**（§5.4） | ProTable 适配层做换算 |
| React 18 / Vite 5 | README 定 **React 19 / Vite 7** | 以 README 为准 |

> **判断准则**：视觉（颜色/布局/质感/交互动线）可大胆复用 Aether；**任何涉及"有哪些字段、哪些表、返回什么结构、有哪些错误"的，一律回到 `janus-server` 源码核对**，不信原型稿。

## 11. 后端暂不支持、待后续迭代的功能（必读）

> 与 §10 不同：§10 是**永久裁剪**（不属于本平台/属 Hermes 端，永远不做）；本节是**在范围内、但后端当前未实现**的能力——Aether 设计了、产品要做，但后端 API 还没有。前端**当前迭代不要依赖这些**，需求方/后端排期后再接入。
>
> 本节结论全部来自对 `janus-server/src` 的逐模块源码核对（admin 各模块 router/schemas + db/models + tasks/registry）。每条都标注了**后端现状**与**前端当前处置**。

### 11.1 缺口总览

| Aether 功能 | 后端现状（源码核对） | 缺什么 | 前端当前处置 |
|---|---|---|---|
| **Dashboard 概览 KPI/聚合** | 仅 `usage/stats`（总请求/总 token/总成本/平均延迟/错误率/成败计数，全局或按 user/model 过滤），无时间维度 | 今日 KPI 计数、**环比趋势**、**7 日趋势序列**、**活跃用户去重(HLL)**、按模型/部门**分布**、**渠道负载 %** | 概览页**降级实现**：用 `usage/stats` 拼当前可得的卡片；趋势/分布/环比**暂缓**或标「即将上线」 |
| **系统健康条**（gateway/db/redis/wecom/scheduler 在线状态） | 仅有**渠道**健康（`catalog/channels/{id}/health`，基于错误率/探针），无平台组件级健康聚合端点 | gateway/db/redis/企微/调度器的 online/degraded/offline 状态 | 健康条**暂缓**；先只做渠道健康（已支持） |
| **配额超限行为** `over_limit_action`（429_reject / **downgrade** / alert_only） | `quota` 只有 `enforce: bool`（开/关硬限），超限恒为硬拒 `quota.exceeded` | downgrade（自动降级到更便宜模型）、alert_only（放行+告警）两种软行为 | 配额表单**只做** enforce 开关 + 三维度（§4.2）；不暴露 action 选择器 |
| **配额实时用量进度**（used/limit 进度条） | `quota` 读模型只有 `limit_value`，**无 used 实时值**字段 | 每条配额的当期 used 实时读出 | 配额列表**不画**进度条；只展示阈值配置。用量另走 `usage` 模块 |
| **渠道连通性测试**（"测试"按钮 → 延迟/错误） | catalog 渠道仅 CRUD + key 轮换 + 健康查询/手动恢复，**无连通性测试端点** | 主动 ping 上游返回延迟/错误的端点 | **不做**测试按钮；渠道状态看健康（错误率/探针） |
| **渠道余额查询**（balance via billing API） | 无 balance 字段、无 billing 端点 | 调上游计费 API 拉余额 | **不展示**余额列 |
| **渠道模型自动发现**（GET /models 同步） | 无 discovery 端点；模型/部署是手工维护 | 拉上游可用模型列表 | 模型/部署**手工录入**，无「同步」按钮 |
| **企微通讯录同步**（用户/部门自动拉取、离职冻结） | `user_oauth` 表 + `OAuthSource.wecom` 枚举存在，但 `tasks/registry` 只有 3 个 job（用量 flush / 日志 flush / 渠道探针），**无目录同步 cron** | 企微拉人/拉部门的同步任务 | 用户/部门**手工 CRUD**；无「同步通讯录」入口 |
| **企微扫码登录** | 有 `user_oauth` 结构基础，但未见扫码登录链路落地 | 企微 OAuth 扫码登录流程 | 登录页**先做密码登录**；扫码入口暂缓 |
| **用户列表内联用量**（daily_used/monthly_used/last_active/可用模型聚合） | `UserRead` 无这些字段（仅身份/部门/状态/角色） | 用户行内的实时用量、最后活跃、模型聚合 | 用户表**不显示**这些列；用量在 `usage` 模块按 user 过滤查看 |
| **在线会话富信息**（client_type/client_version/expires_at/last_active） | `OnlineSessionRead` 仅 `access_jti/user_id/username/ip/user_agent/login_at`（**支持强制下线** ✅） | 客户端类型/版本、过期时间、最后活跃 | 会话表展示**已有字段** + 强制下线；其余列暂缓 |
| **SK-Key 按 Key 限定模型范围** | `ApiKeyCreate` 无 `models` 字段（有 user/name/expires/remark，**创建返回一次明文** ✅） | 每个 key 可调用模型白名单 | 创建/编辑 Key **不做**模型范围选择 |
| **MCP 工具注册表 + 调用统计** | `src/mcp/server.py` 仅 `FastMCP(name="janus")` 协议骨架，**无 mcp_tool/mcp_call_log 表、无注册 CRUD** | 见 §10（**永久裁剪**，工具是代码不是数据） | **不建**（与 §10 一致） |

### 11.2 后端**已支持**、可放心建设的能力（对照确认）

避免误判「后端没有」而漏做——以下经源码确认**已实现**，前端应正常建设：

- **用户 / 部门 / 角色 / 菜单（RBAC）**：`users`、`departments`、`roles`、`menus` 全 CRUD；`role`/`menu`/`user_role`/`role_menu` 表齐全，权限码 `mod:res:act` 生效。
- **登录日志**：`login_log` 表存在（`username`/`failure_reason`/`AuditOutcome`，按 user/username 建索引）——**登录日志页可做**。
- **操作审计**：`oper_log` 表 + 审计中间件，写操作留痕。
- **目录链**：`upstream_channel` / `channel_key`（含 key 轮换）/ `logical_model` / `model_deployment` 全 CRUD；**渠道健康**（错误率+探针，含手动恢复）已支持。
- **目录变更日志**：`catalog/changelog`（before/after/diff）已支持。
- **模型分配**：`user_model_grant`，`scope=user/department` + `is_default`，CRUD 齐全。
- **配额**：`scope × scope_id × logical_model_id × period × metric × limit_value × enforce` 的 CRUD（软肋见 11.1）。
- **限流**：`rate_limit_rule` 表 + `rate_limits` 模块。
- **SK-Key**：创建（明文返回一次）/ 改 / 禁用 / 轮换，`key_prefix`/`last_used_at`/`expires_at` 已有。
- **用量明细 + 聚合**：`usage_record` 全字段（prompt/completion/total tokens、cost、status、latency、**`downgraded_features`**）+ `usage/stats` 聚合。
- **网关可观测**：`gateway_request_log`（逐请求：状态码/错误码/延迟/ttft/tokens/缓存命中/重试/fallback）+ 队列健康 + DLQ 查看/清空。**近期异常列表可由此实现**（按 status_code/error_code 过滤）。
- **在线会话**：列出 + **强制下线**（字段精简，见 11.1）。

> **注意一个反直觉点**：`usage_record` 已带 `downgraded_features` 字段，说明**网关侧可能已有某种降级记录能力**；但 §11.1 所指的是**配额配置层的 `over_limit_action`**（管理员可配的超限策略），二者不同——前端配额表单当前仍不暴露 action 选择，待后端补 quota 写模型后再说。

### 11.3 前端落地建议

- **概览页**按「能做的先做」：KPI 卡用 `usage/stats` 能算的先上，趋势/环比/分布/健康条用占位组件 + 「即将上线」标识，后端补端点后替换数据源即可（组件结构提前留好）。
- **不要 mock 假数据冒充真实功能**。缺口能力宁可**显式标注未上线**，也不要用前端假数据制造「已支持」的错觉（这正是 Aether 原型的坑）。
- **codegen 是缺口的天然探针**：`pnpm codegen` 生成的 `generated/` 里若**没有**某端点/字段，即代表后端未支持——以 generated 为准，不要手写类型「补全」一个并不存在的后端能力。
